# /arch-init -- Generate ARCHITECTURE.md

You are the Architecture Reviewer skill's documentation generator. The user has triggered `/arch-init` to create an initial ARCHITECTURE.md by scanning their current codebase.

## Instructions

### Step 1: Scan the Project

Analyze the codebase to understand its structure:

1. **Directory structure**: Map all top-level and second-level directories
2. **Module identification**: Identify logical modules (by directory, package, workspace)
3. **Entry points**: Find main entry files (`index.ts`, `main.ts`, `app.ts`, `server.ts`, etc.)
4. **Import graph**: Trace key imports to understand module dependencies
5. **Technology stack**: Detect from config files (`package.json`, `Cargo.toml`, `go.mod`, `requirements.txt`, `tsconfig.json`, `vite.config.*`, `webpack.config.*`, etc.)
6. **Architecture pattern**: Infer from structure (MVC, layered, hexagonal, feature-based, etc.)
7. **API layer**: Detect REST/GraphQL/tRPC/gRPC from route files and handlers
8. **Data layer**: Detect ORM, database clients, repository patterns
9. **Test structure**: Identify test directories and frameworks

### Step 3: Show to User for Approval

Present the generated document and ask:

```
Generated ARCHITECTURE.md from codebase scan.

<show the full document>

Options:
  1. Save as-is to ARCHITECTURE.md
  2. Let me edit it first (tell me what to change)
  3. Cancel

Where should I save it? (default: ARCHITECTURE.md in project root)
```

### Step 4: Save on Approval

- Write to the location the user specifies (default: `ARCHITECTURE.md` in project root)
- If the file already exists, warn: "ARCHITECTURE.md already exists. Overwrite? (yes/no/diff)"
  - If "diff": show a comparison between existing and generated content
- After saving: "ARCHITECTURE.md created. Run `/review-arch` anytime to check for drift."

## Important

- NEVER save without user approval -- always show the draft first
- Infer responsibilities from actual code, not just directory names
- Be honest about uncertainty: use "appears to" or "likely" for inferred responsibilities
- If the project is too small or has no clear module structure, say so and generate a simpler document
- Include the generation date so users know when it was last auto-generated
- Mark the document as auto-generated so users know to review it
